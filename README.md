## Генерация слотов

<!-- Опишите выбранный подход: lazy vs eager, скользящее окно и т.д. -->

**Подход:** Lazy generation (ленивая генерация по запросу).\
**Принцип работы:** При первом запросе доступных слотов на конкретную дату сервис
проверяет, существуют ли слоты в БД (CountByRoomAndDate). 
Если нет — генерирует 30-минутные слоты на основе расписания
и сохраняет их через BulkCreate. Повторные запросы возвращают уже сохранённые слоты.\
**Обоснование:** \
1. Не нужен фоновый процесс для предварительной генерации
2. Слоты не дублируются, не посылаются доп запросы на повторную генерацию.
3. Оптимизация через COUNT ускорила работу ручки с 18ms до 9ms, что сильно важно,
учитывая, что по ТЗ это самая высоконагруженная ручка.
4. Так как большинство запросов приходится на ближайшие дни, слоты на эти даты быстро появятся в БД
и затем будут предоставляться без генерации.

## Conference Service (доп. задание)

При создании брони с `createConferenceLink: true` сервис обращается к Conference Service по gRPC для получения ссылки на конференцию.

### Обработка сбоев

| Сценарий                              | Поведение                                                                                              |
|---------------------------------------|--------------------------------------------------------------------------------------------------------|
| Conference Service недоступен         | Бронь создаётся, возвращается fallback-ссылка                                                          |
| Таймаут запроса 3 сек                 | 	Аналогично — fallback-ссылка, бронь создаётся успешно                                                 |
